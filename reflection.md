Memory Leak Observations

The memory leak here is in the for loop, which uses malloc() to get memory. In the memory leak demonstration section of the for loop, each iteration is trying to allocate memory for temp_item. The difference here is that there is no free statement that frees this memory. A memory leak occurs when the application allocates memory that has no reference and cannot be freed.

If this code were looped 1000 times, the memory usage would keep on increasing. It could increase for many loops, resulting in a huge amount of memory being wasted. In a short-running application like a command-line tool, the amount of memory wasted would be relatively small, and so it may not matter. However, in a long-running application like a server, the amount of memory wasted can be severe, and after hours or even days of increased memory usage, the system can become swamped with little memory remaining for normal operation. High swap usage, system lag, and application crashes would then often follow. Memory leaks are a very serious issue in production systems.

Bugs in this code: Free() is called inside the loop. This bug is present to show the memory leak that can occur due to not freeing the memory allocated to a variable using malloc(). The memory allocated by malloc() has to be freed by calling free() in order to avoid a memory leak. If the free() is not called, then the memory can start leaking, and this memory leak can start causing problems when the amount of memory required to hold the data of the nodes in the tree increases. The free() should be called inside the loop in order to match the malloc() called inside the loop. This way, the memory usage of a program in C is kept stable by matching every malloc() call with a free() call.

Python vs C Memory Management

One of the basics of Python is that memory management is handled internally, and this makes coding easier and less error-prone. We can create a list and add elements to it very easily, and Python will take care of the memory management. It will allocate memory when needed and deallocate it when an object is no longer needed. When I was coding in C, I had to allocate memory to the variables that I was using with the malloc() function, check if the memory was allocated or not, increase the memory allocated to a variable with realloc(), and free the memory allocated with the free() function. I had to be careful not to have memory leaks, I had to be careful not to have dangling pointers, and I had to be careful not to have invalid memory access.

Python uses reference counting for memory management. Reference counting is a simple type of garbage collection algorithm. Every object has a reference count. When a new reference to an object is created, its reference count is incremented. When a reference to an object is deleted, its reference count is decremented. When an object’s reference count reaches zero, it is considered garbage and its memory can be freed. In addition to reference counting, Python also has a garbage collector that can reclaim objects involved in a reference cycle. A reference cycle is a situation in which two or more objects reference each other but are not referenced from the call stack.

The great majority of memory management is done by Python. However, a number of relatively rare situations involve a large amount of low-level memory activity. A non-exhaustive list of such situations includes memory allocation and garbage collection, as well as tracking references in order to identify garbage (unused) objects. In C, one is responsible for all memory management. It allows for precise control of the memory and, therefore, high performance. On the other hand, it is very easy to commit errors.

Garbage Collection Trade-offs

A program implemented in Python will spend more time on garbage collection. A garbage collector records references to memory locations and monitors memory locations with no references. Therefore, a garbage collector can waste CPU time and memory space. The garbage collection can be very slow. This is not very suitable for a real-time application.

One downside of this approach is that memory leaks and memory overflow are possible. Although garbage collection is a common feature in many programming languages (such as Java), this is not always the first thing we worry about. Garbage collection does make our coding work much simpler. However, for systems programming, such as writing an operating system or an embedded system, low latency and high performance are very important. That is why systems programmers almost always choose to code in C, rather than in other programming languages like Java or C#. C gives more control over the memory of a program. It can be easier to create memory leaks and memory overflow problems, but it also means the program will never be paused due to garbage collection.
